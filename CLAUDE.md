# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

- `npm install` — install dependencies
- `npm start` — launch the app (`electron . --no-sandbox`; the flag is required on this Linux setup to avoid sandbox permission errors)
- `npm run dist` — produce a Linux release via `electron-builder` (AppImage + `.deb`) into `release/`. Build config lives in the `"build"` field of `package.json`. `electron` must stay in `devDependencies` (electron-builder refuses to package if it's in `dependencies`).
- There is no lint or test setup. The `test` script in `package.json` is a placeholder. `test-parse.js` is a one-off `music-metadata` smoke check (`node test-parse.js`), not a test runner.

### Release process

GitHub releases combine local Linux artifacts with macOS artifacts produced by CI:

1. Bump `version` in `package.json`.
2. `npm run dist` locally — produces `release/Audex-<v>.AppImage` and `release/audex-player_<v>_amd64.deb`.
3. Commit, then `git tag -a v<v> -m "..."` and `git push origin main v<v>`.
4. `gh release create v<v> release/Audex-<v>.AppImage release/audex-player_<v>_amd64.deb --title "..." --notes "..."`.
5. The tag push triggers `.github/workflows/release.yml`, which runs two parallel jobs:
   - **`build-macos`** on `macos-latest` → `.dmg` and `.zip` for x64 and arm64.
   - **`build-windows`** on `windows-latest` → NSIS installer (`.exe`) and `.zip` for x64.
   Both jobs attach artifacts to the release matched by tag (`softprops/action-gh-release`).

Builds are **unsigned**:
- macOS (`CSC_IDENTITY_AUTO_DISCOVERY=false`, `mac.identity: null`): users have to bypass Gatekeeper manually (right-click → Open, or `xattr -d com.apple.quarantine`).
- Windows: SmartScreen will warn until the binary gains reputation; users click "More info → Run anyway".
Release notes should mention both. The `release/` directory is gitignored.

## Architecture

Electron app, three processes wired through `contextBridge`:

- **`main.js`** (main process) — owns the `BrowserWindow`, file dialogs, and the recursive directory walker `scanDir`. All filesystem and tag I/O lives here behind `ipcMain.handle(...)`. The MP3 tag writer (`music:writeMetadata`) explicitly refuses non-MP3 paths because `node-id3` is ID3v2-only. `app.setName('Audex')` is called at module load so the MPRIS identity (visible in the GNOME top bar) matches the brand instead of falling back to the npm package name.
- **`preload.js`** — thin bridge that re-exports the IPC handlers as `window.electronAPI.{openFiles, chooseFolder, scanFolder, parseMetadata, writeMetadata, revealInFolder}`. `contextIsolation: true` and `nodeIntegration: false`, so the renderer must go through this API for anything privileged. Note `webSecurity: false` in `main.js` — it is what allows the renderer to load `file://` paths into `<audio>` and into `background-image: url(...)` for covers.
- **`renderer.js`** — single 1100-line module holding *all* UI state and event wiring. There is no module system, bundler, or framework. Append new functionality to the existing section comments (`// ── Section ──`) rather than introducing new files unless the user asks.

### Renderer state model

All state is module-level `let` bindings at the top of `renderer.js`:

- `library` — the in-memory array; each element is the object returned by `music:parseMetadata` (with `cover` possibly populated from `coverCache`).
- `libraryMeta` — the same shape minus `cover`, used for persistence.
- `favorites`, `playlists`, `recents`, `settings` — persisted to `localStorage`.
- `currentQueue` / `currentTrackIndex` — `currentTrackIndex` is an index into `library` (not into the queue). `nextTrack` / `prevTrack` translate it through `currentQueue.findIndex(...)`. Preserve this invariant when touching playback.

### Persistence rules

- All persistent state goes through `localStorage` via the helpers in the `// ── Persistence ──` section (`saveLibrary`, `savePlaylists`, `saveSettings`, `saveRecents`). Never call `localStorage.setItem` directly elsewhere.
- **Storage-key prefixes are intentionally mixed**: `ambevor-library-meta` and `ambevor-favorites` keep the legacy prefix from the previous app name so existing users' data survives the Audex rebrand. New keys use the `audex-` prefix. Do not "fix" these — see the `LS` constants at the top of `renderer.js`.
- Cover art is deliberately **not** stored in `localStorage` (quota). `saveLibrary` strips `cover` before serializing. On boot, `restoreCovers()` re-parses every track via IPC to repopulate `coverCache`. Any code path that adds a track must also write to `coverCache[path]` if it has a cover.

### UI rendering

- DOM is generated by string templates in render functions (`renderLibrary`, `renderFavorites`, `renderPlaylistDetail`, etc.). Anything user-supplied that enters innerHTML must go through `escapeHtml(...)`.
- After any state mutation that affects rows, call `refreshCurrentViewRows()` — it dispatches to the correct re-render based on `currentView`. Update `renderCounts()` separately when library/playlist/favorites counts change.
- `refreshPlayingHighlight()` is the cheap path that re-paints the "now playing" indicator without rebuilding rows. It handles virtual-list views (library / favorites / playlist-detail) via `VList.refreshVisible()` and the artist-detail view via `refreshPlainListHighlight()` (which patches DOM rows directly because artist-detail doesn't use a virtual list). Any new track-list view must be wired into one of those two branches or its `.playing` highlight will go stale on track change.
- The fullscreen overlay and the mini-player render the *same* now-playing state. `updateNowPlayingUI`, `updatePlayButtonUI`, `updateFavoriteUI`, `updateShuffleUI`, `updateRepeatUI` each update both surfaces; keep new now-playing UI dual-targeted the same way.
- Icons are inline `<symbol>` definitions in `index.html` referenced via `<use href="#i-...">`. Add new icons to that defs block; don't pull in external SVGs.

### System integration (Linux / MPRIS)

- Now-playing metadata and transport controls in the GNOME top bar are wired via the standard `navigator.mediaSession` API in `renderer.js` (Chromium maps it to MPRIS on Linux). `updateMediaSessionMetadata` is called from `updateNowPlayingUI`; `playbackState` is synced from `updatePlayButtonUI`. Action handlers (`play` / `pause` / `previoustrack` / `nexttrack` / `seekto`) are registered once at boot. If you add a new now-playing surface, keep this triple-targeted (mini-player + fullscreen + MediaSession).

### In-development flags

- The **Downloads** section is intentionally inert: the settings toggle is rendered with the `is-disabled` class and an "в разработке" badge, and `renderer.js` force-sets `settings.downloads = false` on boot so any previously-stored "on" value is ignored. When the feature ships, remove the force-reset and drop `is-disabled` from the toggle markup.

### Theming

- Single source of truth: `data-theme="dark|light"` on `<html>`, with CSS variables in `style.css`. `applyTheme` resolves `'system'` against `prefers-color-scheme`. Don't hard-code colors in JS or HTML — read from CSS vars where dynamic styling is needed.

### Languages

- UI strings in `index.html` and `renderer.js` are Russian (the user is Russian-speaking). Code, comments, and identifiers are English. Keep this split when adding features. The README is English.

## Conventions to preserve

- Vanilla everything — no bundler, no TypeScript, no UI framework. Don't introduce one unless the user asks.
- File paths from the OS are passed around as-is and prefixed with `file://` only at the point they hit `<audio>` or `background-image`. Don't pre-encode them elsewhere.
- The `mp3-player/` directory in the repo root is unrelated scratch content and is not part of the app — ignore it unless the user references it.
